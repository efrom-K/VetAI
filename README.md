## 📋 Overview
**VetAI** — это высокотехнологичная экосистема для предиктивной аналитики в ветеринарии. Мы разрабатываем «умное ядро», которое обучается на миллионах синтезированных и реальных клинических случаев, чтобы обеспечить врачей точным диагностическим вектором.

Ядро **Neural Kernel v15_opt** использует гибридную архитектуру: сочетание глубоких нейронных сетей (**Deep Learning**) на базе **TensorFlow** и методов градиентного бустинга для интерпретации табличных данных.

## 📊 Dataset: Expert-Curated Medical Data

Одной из главных инноваций VetAI является его **Data-Origin**. В отличие от моделей, обученных на неструктурированных данных из открытых источников, наше ядро базируется на:

1. **Expert Synthesis:** База данных формировалась на протяжении длительного времени практикующими врачами.
2. **Clinical Accuracy:** Каждый паттерн в `DISEASE_PROFILES` прошел верификацию на соответствие протоколам доказательной медицины.
3. **Structured Knowledge:** Перевод многолетнего клинического опыта в машиночитаемые CSV-матрицы с учетом тончайших нюансов патофизиологии.

## 🔬 Architecture: Neural Kernel v15
Архитектура v15 построена на принципе **Biologically Constrained ML** (Машинное обучение с биологическими ограничениями).

1. **Feature Engineering & Latent Space** Модель обрабатывает более 200 входных параметров, включая разреженные категориальные данные (**One-Hot Encoded**). Использование `protobuf==3.20.0` обеспечивает быструю сериализацию данных при обмене между компонентами системы.

2. **Explainable AI (SHAP)** В ветеринарии важен не только результат, но и аргументация. Интеграция SHAP позволяет визуализировать вклад каждого симптома в итоговое решение:
   * *Пример: Модель подсвечивает, что «Живот увеличен» внес +40% к вероятности ГРЖ, а «Нормальная температура» снизила вероятность инфекции на 15%.*

3. **Data Synthesis Protocol** Обучение проводится на массиве данных в **1,500,000 записей**, базирующихся на авторских медицинских профилях, **вручную составленных Ильей Свечниковым**. Это позволяет модели знать редчайшие патологии, с которыми врач встречается раз в 10 лет.

## 📄 Reporting tools

В систему интегрирован модуль генерации ветеринарных протоколов, который позволяет мгновенно превратить результаты анализа в документ, готовый к печати.

### Основные возможности:
* **Автоматическое формирование:** Протокол включает в себя диагноз, расчетную вероятность, полный анамнез и список выявленных симптомов.
* **Поддержка кириллицы:** Использование шрифтов DejaVuSans гарантирует корректное отображение русского текста.
* **Модульная архитектура:** Логика формирования PDF вынесена в отдельный сервис `pdf_generator.py`. Это позволяет:
    * Легко менять дизайн отчета, не затрагивая код нейросети.
    * Добавлять новые поля (результаты анализов, данные тонометрии и т.д.) простым обновлением словаря данных.
    * Масштабировать отчет под разные форматы (A4, мобильный чек и др.).

### Технический стек экспорта:
* **FPDF2:** Библиотека для генерации PDF (выбрана за высокую скорость работы в Python-среде).
* **Byte-streaming:** Документ генерируется "на лету" в оперативной памяти и передается пользователю без создания временных файлов на сервере, что повышает безопасность и производительность.

## 📡 RESTful API Interface

Проект VetAI предоставляет стандартизированный программный интерфейс на базе фреймворка FastAPI, реализующий архитектуру «Model-as-a-Service» (MaaS). Это позволяет инкапсулировать логику нейросетевой диагностики и предоставлять доступ к предиктивным мощностям внешним потребителям: мобильным клиентам, CRM-системам ветеринарных клиник или интеграционным шинам.

### Ключевые особенности реализации:
* **Asynchronous Processing:** Использование ASGI-сервера (Uvicorn) обеспечивает высокую пропускную способность и эффективную обработку конкурентных запросов, что критично при развертывании на энергоэффективных серверах.
* **Automatic Documentation:** Интеграция со спецификацией OpenAPI (Swagger) и ReDoc позволяет проводить мгновенное тестирование эндпоинтов и автоматизировать генерацию клиентских SDK.
* **Data Validation:** Строгая типизация входящих данных через модели Pydantic исключает передачу некорректных векторов признаков на вход нейросети.
* **Hardware Acceleration:** API поддерживает прозрачный проброс аппаратных ресурсов (CUDA) при работе в контейнеризированном окружении через nvidia-container-runtime.

| Метод | Эндпоинт | Описание |
|:---|:---|:---|
| **GET** | / | Health-check: мониторинг состояния сервиса и версии развернутой модели. |
| **POST** | /predict | Инференс: обработка вектора симптомов и возврат топ-N диагнозов с весами вероятности. |
| **GET** | /docs | Интерактивная документация Swagger UI для отладки. |

### Интеграционный пример (Payload):

Запрос в формате JSON должен содержать словарь symptoms, сопоставленный с признаками из обучающей выборки (TRAINING_FEATURES):

## 🚀 Deployment & Development
### Local Setup via Docker
Для запуска среды разработки со всеми зависимостями:

```bash
# Сборка образа
docker build -t vet-ai-image . --no-cache

# Запуск с поддержкой СPU единого стека APP+API
docker run -it -p 8501:8501 -p 8000:8000 --rm `
    -v "${pwd}:/app" `
    vet-ai-image `
    sh -c "uvicorn api:app --host 0.0.0.0 --port 8000 & streamlit run app.py --server.port 8501"

# Запуск с поддержкой GPU единого стека APP+API
docker run --gpus all -it -p 8501:8501 -p 8000:8000 --rm `
    -v "${pwd}:/app" `
    vet-ai-image `
    sh -c "uvicorn api:app --host 0.0.0.0 --port 8000 & streamlit run app.py --server.port 8501"

# Запуск бенчмарка системы
docker run --gpus all --rm -v ${pwd}:/app vet-ai-image python benchmark.py

```

**Download the weight file from the Releases section and place it in the root folder.**

### CI/CD Pipeline
Проект готов к масштабированию через **Kubernetes (K8s)**. Модульная структура `app.py` и использование `joblib` для версионирования весов позволяют проводить **A/B тестирование** различных версий ядра v15.1, v15.2 параллельно.

## 📊 Scientific Methodology
В основе **VetAI** лежит научная работа кафедр Росбиотеха и экспертный опыт врачей. Мы трансформируем классические диагностические справочники в математические модели.

* **Precision:** `0.94+` на валидационных сетах.
* **Recall:** `0.91+` (критически важно для исключения ложноотрицательных результатов в экстренной хирургии).
* **Core Inference Latency:** `0.0695ms` Это чистое время, затрачиваемое моделью на выполнение математических операций внутри GPU, актуально для NVIDIA RTX 30-series
* **End-to-End System Latency:** `58ms` Время от момента нажатия кнопки пользователем до вывода результата на экран.

## 🛠 Engineering & Performance Optimization

В версии v15.1 был проведен комплекс мероприятий по глубокой оптимизации вычислительного конвейера, направленный на минимизацию задержек (latency) и повышение отказоустойчивости системы.

### 1. Вычислительная оптимизация (Mixed Precision)

Для эффективного использования аппаратных ресурсов GPU (архитектуры NVIDIA Ampere и Turing) внедрена технология смешанной точности (Mixed Precision).

* **Механизм:** Большинство тензорных операций переведено в формат float16, в то время как критические узлы (финальный слой Softmax) сохранены в float32 для обеспечения численной стабильности.

* **Результат:** Активация ядер Tensor Cores позволила достичь удельной производительности в 0.107 ms на одну операцию при пакетной обработке (Batch Inference), что делает систему пригодной для обработки Big Data.

### 2. Оптимизация среды исполнения (Runtime Efficiency)

Проведено исследование влияния среды виртуализации на производительность инференса.

* **Native Linux vs WSL2:** Установлено, что отсутствие абстракций гипервизора в нативной среде Linux сокращает задержку обработки на CPU в 2.4 раза (с 255 ms до 106 ms).

* **Instruction Set Adaptation:** Алгоритм оптимизирован под наборы инструкций AVX, что гарантирует стабильную работу на Legacy-оборудовании (протестировано на архитектуре Intel Sandy Bridge) без поддержки современных расширений AVX2/AVX-512.

### 3. Рефакторинг сериализации и CI/CD

Для устранения несовместимости механизмов LossScaleOptimizer и формата HDF5 в современных версиях TensorFlow, была изменена логика сохранения артефактов.

* **Persistence Strategy:** Реализована схема экспорта весов с исключением графа оптимизатора (include_optimizer=False).

* **Контейнеризация:** Конфигурация Docker-образа унифицирована для бесшовного проброса GPU-интерфейсов через NVIDIA Container Toolkit, обеспечивая автоматическое согласование CUDA-драйверов хоста и гостевой ОС.
